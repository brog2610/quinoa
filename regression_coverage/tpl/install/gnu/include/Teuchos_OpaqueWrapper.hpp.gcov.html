      65                 :            :  *
      66                 :            :  * If you want to create an RCP to an opaque handle (an instance of a
      67                 :            :  * type like MPI_Comm), use the opaqueWrapper() nonmember template
      68                 :            :  * function.  If that opaque handle needs to be freed after all
      69                 :            :  * references to it go away, then supply a "free" function to
      70                 :            :  * opaqueWrapper().  The type returned by opaqueWrapper() is
      71                 :            :  * <tt>RCP<OpaqueWrapper<T> ></tt>, an RCP (reference-counted "smart"
      72                 :            :  * pointer) to a wrapper of the opaque handle type T.  Users are not
      73                 :            :  * allowed to construct an OpaqueWrapper object explicitly.  You must
      74                 :            :  * use the opaqueWrapper() nonmember function to do so.
      75                 :            :  *
      76                 :            :  * \section Teuchos_OpaqueWrapper_Prereq Prerequisites
      77                 :            :  *
      78                 :            :  * In order to understand this documentation, you must first have
      79                 :            :  * learned how to use RCP (Teuchos' reference-counted "smart" pointer
      80                 :            :  * class) to manage dynamically allocated memory and other resources.
      81                 :            :  * It also helps to be familiar with MPI (the Message Passing
      82                 :            :  * Interface for distributed-memory parallel programming), but this is
      83                 :            :  * not required.
      84                 :            :  *
      85                 :            :  * \section Teuchos_OpaqueWrapper_Handles What are opaque handles?
      86                 :            :  *
      87                 :            :  * Many different software libraries use the <i>opaque handle</i>
      88                 :            :  * (a.k.a. opaque object) idiom to hide the internals of a data
      89                 :            :  * structure from users.  This standard technique allows users to
      90                 :            :  * treat an instance of a data structure as a handle.  Users may pass
      91                 :            :  * the handle around as if it were a simple value type (like int), and
      92                 :            :  * must call nonmember functions in order to create, operate on, use,
      93                 :            :  * or destroy instances of the data structure.  The MPI (Message
      94                 :            :  * Passing Interface) standard is full of examples of the opaque
      95                 :            :  * handle idiom, including MPI_Comm (for communicators), MPI_Datatype
      96                 :            :  * (for standard and custom data types), and MPI_Op (for standard and
      97                 :            :  * custom reduction operations).
      98                 :            :  *
      99                 :            :  * In general, opaque handles (corresponding to the Opaque template
     100                 :            :  * parameter) must be <i>assignable</i>.  This means that copy
     101                 :            :  * construction and assignment (operator=) must be syntactically
     102                 :            :  * correct for instances of Opaque.  This is certainly true of MPI's
     103                 :            :  * opaque handle types.
     104                 :            :  *
     105                 :            :  * Opaque handles are a useful technique, but they interfere with
     106                 :            :  * correct use of reference-counted "smart" pointer types such as
     107                 :            :  * Teuchos' RCP or std::shared_ptr.  We will explain below why this is
     108                 :            :  * the case.  The OpaqueWrapper base class allows opaque handles to be
     109                 :            :  * wrapped by a real object, whose address you can take.  This is
     110                 :            :  * needed in order to wrap an opaque object in a RCP, for example.
     111                 :            :  *
     112                 :            :  * \section Teuchos_OpaqueWrapper_Special Why do opaque handles need special treatment?
     113                 :            :  *
     114                 :            :  * The OpaqueWrapper class was motivated by MPI's common use of the
     115                 :            :  * opaque handle idiom.  For MPI, passing MPI_Comm, MPI_Datatype, and
     116                 :            :  * MPI_Op objects around by handles hides implementation details from
     117                 :            :  * the user.  Handles also make it easier to access MPI functionality
     118                 :            :  * from Fortran, so that C, C++, and Fortran can all share the same
     119                 :            :  * handle mechanism.  In fact, some MPI implementations (such as
     120                 :            :  * MPICH, at least historically if not currently) simply implement
     121                 :            :  * these handles all as integers.  (As the MPI standard's advice to
     122                 :            :  * implementers suggests, such an implementation would likely maintain
     123                 :            :  * a table for each MPI process that maps the integer value to a
     124                 :            :  * pointer to the corresponding object.)  For example, MPI_Comm might
     125                 :            :  * be a typedef to int, and MPI_COMM_WORLD might be a C preprocessor
     126                 :            :  * macro for a literal integer value:
     127                 :            :  *
     128                 :            :  \code
     129                 :            :  typedef int MPI_Comm;
     130                 :            :  #define MPI_COMM_WORLD 42
     131                 :            :  \endcode
     132                 :            : 
     133                 :            :  * In this case, the expression <tt>rcp(&MPI_COMM_WORLD)</tt> would
     134                 :            :  * not even compile, since one cannot take the address of an integer
     135                 :            :  * literal such as 42.  (Remember that preprocessor macros get
     136                 :            :  * replaced with their values before the C++ compiler does its work.)
     137                 :            :  * To make this expression compile, one might try the following:
     138                 :            : 
     139                 :            :  \code
     140                 :            :  // THIS FUNCTION IS WRONG.  IT MAY SEGFAULT.
     141                 :            :  Teuchos::RCP<MPI_Comm> getMpiCommPtr()
     142                 :            :  {
     143                 :            :    MPI_Comm comm = MPI_COMM_WORLD;
     144                 :            :    // WRONG!!!  comm is a stack variable!
     145                 :            :    return Teuchos::rcp (&comm, false);
     146                 :            :  }
     147                 :            :  \endcode
     148                 :            : 
     149                 :            :  * Using the returned communicator would result in undefined behavior,
     150                 :            :  * which in practice might be a segfault, memory corruption, or MPI
     151                 :            :  * getting severely confused.  This is because the stack variable
     152                 :            :  * <tt>comm</tt>, which may be just an integer, disappears at the end
     153                 :            :  * of the function.  Its address would no longer point to valid memory
     154                 :            :  * after the function returns.
     155                 :            :  *
     156                 :            :  * The following code is syntactically correct, but may leak memory:
     157                 :            : 
     158                 :            :  \code
     159                 :            :  // THIS CODE LEAKS MEMORY FOR GENERAL MPI_Comm OBJECTS.
     160                 :            :  Teuchos::RCP<MPI_Comm> getMpiCommPtr (MPI_Comm comm)
     161                 :            :  {
     162                 :            :    MPI_Comm *pComm = new MPI_Comm (comm);
     163                 :            :    // Works for comm==MPI_COMM_WORLD or MPI_COMM_SELF;
     164                 :            :    // leaks memory for user-created MPI_Comm objects.
     165                 :            :    return Teuchos::rcp (pComm);
     166                 :            :  }
     167                 :            :  \endcode
     168                 :            : 
     169                 :            :  * The above implementation of getMpiCommPtr() is correct only for the
     170                 :            :  * standard MPI_Comm objects provided by MPI, like MPI_COMM_WORLD and
     171                 :            :  * MPI_COMM_SELF.  It is <i>not</i> correct, and in fact may leak
     172                 :            :  * memory, for custom MPI_Comm objects that the user creates by
     173                 :            :  * calling functions like MPI_Comm_split().  This is because
     174                 :            :  * user-created MPI_Comm objects must be freed by MPI_Comm_free().
     175                 :            :  * Other kinds of opaque objects, like MPI_Datatype and MPI_Op, have
     176                 :            :  * their own free functions.  Thus, even if opaque handles have the
     177                 :            :  * type integer, they really behave like pointers or references.  Some
     178                 :            :  * of them can and should be freed at the end of their useful lives;
     179                 :            :  * others must not.  (Compare std::ostream; std::cout should never be
     180                 :            :  * closed by typical user code, but an output file should be closed.)
     181                 :            :  *
     182                 :            :  * \section Teuchos_OpaqueWrapper_How How to use OpaqueWrapper
     183                 :            :  *
     184                 :            :  * We fix this problem by providing the OpaqueWrapper template base
     185                 :            :  * class and the opaqueWrapper() nonmember template function.  Use
     186                 :            :  * this function to wrap an opaque handle (like an MPI_Comm) in an
     187                 :            :  * RCP.  This ensures that the RCP does the right thing in case the
     188                 :            :  * handle must be freed.  For example, to wrap MPI_COMM_WORLD in a
     189                 :            :  * RCP, just do this:
     190                 :            : 
     191                 :            :  \code
     192                 :            :  RCP<OpaqueWrapper<MPI_Comm> > comm = opaqueWrapper (MPI_COMM_WORLD);
     193                 :            :  \endcode
     194                 :            : 
     195                 :            :  * If you instead want to create a custom MPI_Comm using a function
     196                 :            :  * like MPI_Comm_split(), then you may wrap it in an RCP as follows
     197                 :            :  * (please see discussion later about MPI_Comm_free()):
     198                 :            : 
     199                 :            :  \code
     200                 :            :  MPI_Comm rawComm;
     201                 :            :  // We omit all arguments but the last of MPI_Comm_split, for clarity.
     202                 :            :  int errCode = MPI_Comm_split (..., &rawComm);
     203                 :            :  if (errCode != MPI_SUCCESS) {
     204                 :            :    // ... Handle the error ...
     205                 :            :  }
     206                 :            :  RCP<OpaqueWrapper<MPI_Comm> > comm = opaqueWrapper (rawComm, MPI_Comm_free);
     207                 :            :  \endcode
     208                 :            : 
     209                 :            :  * The optional second argument to opaqueWrapper() is a "free"
     210                 :            :  * function.  It has type OpaqueFree which is a template parameter.
     211                 :            :  * If the free function is provided, then when the RCP's reference
     212                 :            :  * count goes to zero, that function is called to "free" the handle.
     213                 :            :  * If opaqueFree is a free function, then the following must be
     214                 :            :  * syntactically valid, where <tt>opaque</tt> has type <tt>Opaque</tt>:
     215                 :            : 
     216                 :            :  \code
     217                 :            :  opaqueFree (&opaque);
     218                 :            :  \endcode
     219                 :            : 
     220                 :            :  * The function's return value, if any, is ignored.  Furthermore, the
     221                 :            :  * OpaqueFree type must be copy constructible.  (A function pointer is
     222                 :            :  * trivally copy constructible.)
     223                 :            :  *
     224                 :            :  * Users are responsible for knowing whether to provide a free
     225                 :            :  * function to opaqueWrapper().  In this case, because we created an
     226                 :            :  * MPI_Comm dynamically using a communicator "constructor" function,
     227                 :            :  * the MPI_Comm must be "freed" after use.  RCP will automatically
     228                 :            :  * call the "free" function once the reference count of \c comm
     229                 :            :  * reaches zero.
     230                 :            :  *
     231                 :            :  * \note The above example is only correct if the reference count of
