00004   \author    J. Bakosi

00007   \brief     Operations on numbers of scalar components of systems of equations
00008   \details   Operations on numbers of scalar components of systems of equations,
00009     e.g. multiple equation sets of a physical model or a set of equations of
00010     different kinds.
00011 
00012     _Problem:_ We are given a type that is a tk::tuple::tagged_tuple that
00013     contains an arbitrary number of std::vectors of integers. The number of
00014     vectors are fixed at compile-time (accessed via tags) but their (differing)
00015     length is only known at run-time (after parsing user input). What we need
00016     are functions that operate on this data structure and return, e.g., the
00017     total number of components in the whole system or the offset in the whole
00018     data structure for a given tag. The functions should thus be able to operate
00019     on a list of types, i.e., a double for loop over all tags and associated
00020     vectors - one at compile-time and the other one at run-time.
00021 
00022     _Example:_ An example is to define storage for systems of stochastic
00023     differential equations as in walker::ctr::ncomps. In
00024     [walker](walker.html) various types of stochastic differential
00025     equations are implemented and numerically integrated in time in order to
00026     compute and learn about their statistical behavior. Since different types of
00027     systems can be parameterized differently, there can be several systems of
00028     the same type (the number is user-defined so only known at run-time). For
00029     example, it is possible to numerically integrate two systems of Dirichlet
00030     equations, see DiffEq/Dirichlet.h, one with 4, the other one with 5 scalar
00031     variables, parameterized differently, and estimate their arbitrary coupled
00032     statistics and joint PDFs. This requires that each type of system
00033     has a vector of integers storing the number of scalar variables.
00034 
00035     _Solution:_ Looping through elements of a tuple is done via [Boost]
00036     (http://www.boost.org)'s [MetaProgramming Library (MPL)]
00037     (http://www.boost.org/doc/libs/release/libs/mpl)'s [for_each]
00038     (http://www.boost.org/doc/libs/release/libs/mpl/doc/refmanual/for-each.html).
00039     Such operations on types happen at compile-time, i.e., the code runs inside
00040     the compiler and only its result gets compiled into code to be run at
00041     run-time. Advantages are abstraction and generic code that is independent
00042     of the size and order of the tags in the tuple (and the associated vectors).
00043     Though it is not of primary concern for this data structure, this solution
00044     also generates efficient code, since part of the algorithm (for computing
00045     the offset, the total number of components, etc.) runs inside the compiler.
00046     All of this is kept generic, so it does not need to be changed when a new
00047     pair of tag and system of equations are added to the underlying tuple.
00048     Thus, _the main advantage is that changing the order of the vectors in the
00049     tuple or adding new ones at arbitrary locations will not require change to
00050     client-code._
00051 
00052     _An alternative approach:_ Of course this could have been simply done with a
00053     purely run-time vector of vector of integers. However, that would not have
00054     allowed a tag-based access to the different systems of equations. (Sure, one
00055     can define an enum for equation indexing, but that just adds code to what is
00056     already need to be changed if a change happens to the underlying data
00057     structure.) As a consequence, a component in a vector of vector would have
00058     had to be accessed as something like, ncomps[2][3], which is more
00059     error-prone to read than ncomps< tag::dirichlet >( 3 ). Furthermore, that
00060     design would have resulted in double-loops at run-time, instead of letting
00061     the compiler loop over the types at compile-time and do the run-time loops
00062     only for what is only known at run-time. Additionally, and most importantly,
00063     _any time a new system is introduced (or the order is changed), all
00064     client-code would have to be changed._
00065 
00066     For example client-code, see walker::Dirichlet::Dirichlet() in
00067     DiffEq/Dirichlet.h, or walker::Integrator::Integrator in Walker/Integrator.h.
00068 */

00089 //! Map associating a dependent variable of a differential equation to its
00090 //! offset in the particle data array. We use a case-insensitive character
00091 //! comparison functor for the offset map, since the keys (dependent variables)
00092 //! in the offset map are only used to indicate the equation's dependent
00093 //! variable, however, queries (using std::map's member function, find) can be
00094 //! fired up for both ordinary and central moments (which are denoted by upper
00095 //! and lower case, characters, respectively) for which the offset (for the same

00099 //! Number of components storage. All this trickery with boost::mpl allows the
00100 //! code below to be generic. As a result, adding a new component requires
00101 //! adding a single line (a tag and its type) to the already existing list, see
00102 //! typedefs 'ncomps'. The member functions, doing initialization, computing the
00103 //! number of total components, the offset for a given tag, and computing the
00104 //! offset map, need no change -- even if the order of the number of components
00105 //! change.
00106 template< typename... Tags >
00107 class ncomponents : public tk::tuple::tagged_tuple< Tags... > {

00109   private:
00110     //! Create mpl::list from variadic pack, storing no. of comps with types

00113   public:
00114     //! Remove std::vector< ncomp_type > types, i.e., keep only the tags
00115     using tags = typename
00116       boost::mpl::remove< tagsncomps, std::vector< ncomp_type > >::type;

00118   private:
00119     //! Function object for zeroing all number of components
00120     //! \author J. Bakosi
00121     struct zero {
00122       //! Need to store reference to host class whose data we operate on
00123       ncomponents* const m_host;
00124       //! Constructor: store host object pointer
00125       zero( ncomponents* const host ) : m_host( host ) {}
00126       //! Function call operator templated on the type that does the zeroing
00127       template< typename U > void operator()( U ) {
00128         //! Loop through and zero all elements of the vector for this system
00129         for (auto& c : m_host->template get< U >()) c = 0;
00130       }
00131     };

00133     //! Function object for computing the total number of components (i.e., the
00134     //! sum of all of the number of components)
00135     //! \author J. Bakosi
00136     struct addncomp {
00137       //! Need to store reference to host class whose data we operate on
00138       const ncomponents* const m_host;
00139       //! Internal reference used to return the total number of components
00140       ncomp_type& m_nprop;
00141       //! Constructor: store host object pointer and initially zeroed counter
00142       //! reference
00143       addncomp( const ncomponents* const host, ncomp_type& nprop ) :
00144         m_host( host ), m_nprop( nprop = 0 ) {}
00145       //! Function call operator templated on the type that does the counting
00146       template< typename U > void operator()( U ) {
00147         //! Loop through and add up all elements of the vector for this system
00148         for (const auto& c : m_host->template get< U >()) m_nprop += c;
00149       }
00150     };

00152     //! Function object for computing the offset for a given tag (i.e., the sum
00153     //! of the number of components up to a given tag). This is used to index
00154     //! into the particle array and get to the beginning of data for a given
00155     //! differential equation system.
00156     //! \author J. Bakosi
00157     template< typename tag >
00158     struct addncomp4tag {
00159       //! Need to store reference to host class whose data we operate on
00160       const ncomponents* const m_host;
00161       //! Internal reference used to return the offset for the tag given
00162       ncomp_type& m_offset;
00163       //! Internal storage for the index of a system within systems, i.e., I
00164       //! want the second Dirichlet system: m_c = 1. See also offset().
00165       const ncomp_type m_c;
00166       //! Indicates whether the tag (eq system) was found, so it is time to quit
00167       bool m_found;
00168       //! Constructor: store host object pointer, initially zeroed offset
00169       //! reference, and system index we are looking for
00170       addncomp4tag( const ncomponents* const host, ncomp_type& offset,
00171                     ncomp_type c ) :
00172         m_host( host ), m_offset( offset = 0 ), m_c( c ), m_found( false ) {}
00173       //! Function call operator templated on the type that does the offset
00174       //! calculation
00175       template< typename U > void operator()( U ) {
00176         if (std::is_same< tag, U >::value) {
00177           // Make sure we are not trying to index beyond the length for this U
00178           Assert( m_host->template get<U>().size() >= m_c,
00179                   "Indexing out of bounds in addncomp4tag!" );
00180           // If we have found the tag we are looking for, we count up to the
00181           // given system index (passed in via the constructor) and add those to
00182           // the offset, then quit
00183           for (ncomp_type c=0; c<m_c; ++c)
00184             m_offset += m_host->template get<U>()[c];
00185           m_found = true;
00186         } else if (!m_found) {
00187           // If we have not found the tag we are looking for, we add all the
00188           // number of scalars for that tag to the offset
00189           for (const auto& c : m_host->template get< U >()) m_offset += c;
00190         }
00191       }
00192     };
00193 
00194   public:
00195     //! Default constructor: set defaults to zero for all number of components
00196     //! \author J. Bakosi
00197     ncomponents() { boost::mpl::for_each< tags >( zero( this ) ); }

00218     //! \param[in] depvars Vector of vectors of dependent variables

00220     OffsetMap offsetmap( const std::vector< std::vector< char > >& depvars )
00221     const {
00222       OffsetMap offset;
00223       // This is now simple, as the heavy lifting has already been done by
00224       // extracting the dependent variables to a vector of vectors that is
00225       // passed in. Note that for now, we simply assign linearly increasing
00226       // offsets. This may be changed in the future if/when we decide to
