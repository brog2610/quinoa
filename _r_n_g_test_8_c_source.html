00050 //! Global-scope data. Initialized by the main chare and distibuted to all PEs by
00051 //! the Charm++ runtime system. Though semantically not const, all these global
00052 //! data should be considered read-only. See also http://charm.cs.illinois.edu/
00053 //! manuals/html/charm++/manual.html. The data below is global-scope because they
00054 //! must be available to all PEs which could be on different machines. In a
00055 //! previous non-Charm++ design, most of this data was held at class-level, but
00056 //! since the generators in g_rng must be possible to be called from
00057 //! global-scope, as external generators to TestU01, it is easier to make g_rng
00058 //! global-scope, as well the additional data required to initialize it,
00059 //! contained in g_inputdeck (storing all parsed user input).  This is required
00060 //! for serializing during migration of g_rng across the network.
00061 //!
00062 //! Note that the container (std::map) holding tk::RNG objects uses value
00063 //! semantics which is safer and generally less error-prone than reference
00064 //! semantics. At the same time tk::RNG is used in a polymorphic fashion with
00065 //! various classes that adhere to the concepts required by Concept defined
00066 //! inside tk::RNG. tk::RNG does not define a default, i.e., non-templated
00067 //! constructor, since then the "derived" class object could not be initialized
00068 //! rendering the class tk::RNG empty-constructed, which invites abuse and
00069 //! ill-defined behavior. As such, the "derived" class type comes through the
00070 //! constructors and thus would not be available for a pack/unpack migrator
00071 //! required by Charm++ from within. Templating the class tk::RNG is not an
00072 //! option since then we could not hold tk::RNG objects in a simple std::vector.
00073 //! As a result of the above requirements, the tk::RNG objects in g_rng are
00074 //! migrated (here in global-scope) by reinstantiating RNGStack, which
00075 //! reinstatiates the RNG factory, from which the RNGs selected by the user are
00076 //! instantiated.
00077 //!
00078 //! Note also that RNGFactory associates tk::ctr::RNG ids (enum class values) to
00079 //! function pointers (std::function objects pointing to tk::RNG constructors
00080 //! bound with their arguments). Since function pointers cannot simply be
00081 //! serialized and migrated via the network, they must also be recreated on
00082 //! remote machines. This initial migration of global-scope data is done by the
00083 //! Charm++ runtime once the main chare constructor is finished -- see the
00084 //! RNGTestDriver constructor, which initializes the data required for the
00085 //! migration).

00096 //! Pack/Unpack selected RNGs. This Pack/Unpack method (re-)creates the full RNG
00097 //! stack since it needs to (re-)bind function pointers on different processing
00098 //! elements. Therefore we circumvent Charm's usual pack/unpack for this type,
00099 //! and thus sizing does not make sense: sizing is a no-op. We could initialize
00100 //! the stack in RNGTestDriver's constructor and let this function re-create the
00101 //! stack only when unpacking, but that leads to repeating the same code twice:
00102 //! once in RNGTestDriver's constructor, once here. Another option is to use
00103 //! this pack/unpack routine to both initially create (when packing) and to
00104 //! re-create (when unpacking) the stack, which eliminates the need for
00105 //! pre-creating the object in RNGTestDriver's constructor and therefore
00106 //! eliminates the repeated code. This explains the guard for sizing: the code
00107 //! below is called for packing only (in serial) and packing and unpacking (in
00108 //! parallel).
00109 inline
00110 void operator|( PUP::er& p, std::map< tk::ctr::RawRNGType, tk::RNG >& rng ) {

00115       #endif
00116       g_inputdeck.get< tag::param, tag::rngsse >() );
00117     rng = stack.selected( g_inputdeck.get< tag::selected, tag::rng >() );
00118   }

00145     //! \brief Constructor
00146     //! \details The main chare constructor is the main entry point of the
00147     //!   program, called by the Charm++ runtime system. The constructor does
00148     //!   basic initialization steps, e.g., parser the command-line, prints out
00149     //!   some useful information to screen (in verbose mode), and instantiates
00150     //!   a driver. Since Charm++ is fully asynchronous, the constructor
00151     //!   usually spawns asynchronous objects and immediately exits. Thus in the
00152     //!   body of the main chare constructor we fire up an 'execute' chare,
00153     //!   which then calls back to Main::execute(). Finishing the main chare
00154     //!   constructor the Charm++ runtime system then starts the
00155     //!   network-migration of all global-scope data (if any). The execute chare
00156     //!   calling back to Main::execute() signals the end of the migration of
00157     //!   the global-scope data. Then we are ready to execute the driver. Since
00158     //!   the random number generator test suite is parallel and asynchronous,
00159     //!   its driver fires up additional Charm++ chare objects which then call
00160     //!   back to Main::finalize() at some point in the future when all work has
00161     //!   been finished. finalize() then exits by calling Charm++'s CkExit(),
00162     //!   shutting down the runtime system.
00163     //! \see http://charm.cs.illinois.edu/manuals/html/charm++/manual.html
00164     Main( CkArgMsg* msg )
00165     try :
00166       // Parse command line into m_cmdline using default simple pretty printer
00167       m_cmdParser( msg->argc, msg->argv, tk::Print(), m_cmdline ),
00168       // Create pretty printer initializing output streams based on command line
00169       m_print( m_cmdline.get< tag::verbose >() ? std::cout : std::clog ),
00170       // Create RNGTest driver
00171       m_driver( tk::Main< rngtest::RNGTestDriver >
00172                         ( msg->argc, msg->argv,
00173                           m_cmdline,
00174                           tk::HeaderType::RNGTEST,
00175                           RNGTEST_EXECUTABLE,
00176                           m_print ) ),
00177       m_timer(1)        // Start new timer measuring the total runtime
00178     {
00179       delete msg;
00180       mainProxy = thisProxy;
00181       // Fire up an asynchronous execute object, which when created at some
00182       // future point in time will call back to this->execute(). This is
00183       // necessary so that this->execute() can access already migrated
00184       // global-scope data.
00185       CProxy_execute::ckNew();
00186       // Start new timer measuring the migration of global-scope data
