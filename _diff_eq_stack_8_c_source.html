00057 //  Constructor: register all differential equations into factory
00058 //! \details This constructor consists of several blocks, each registering a
00059 //!   potentially large number of entries in the differential equation factory,
00060 //!   m_factory, which is of type walker::DiffEqFactory, a std::map. At this
00061 //!   time, each type of differential equation can be configured to use a unique
00062 //!   _initialization policy_ and a unique _coefficients policy_. (More types
00063 //!   of policies will most likely come in the future.) The policy classes are
00064 //!   template arguments to the differential equation classes and influence
00065 //!   their behavior in a different way, abstracting away certain functions,
00066 //!   e.g., how to set initial conditions and how to update their coefficients
00067 //!   during time integration. For more information on policy-based design, see
00068 //!   http://en.wikipedia.org/wiki/Policy-based_design. This abstraction allows
00069 //!   [separation of concerns](http://en.wikipedia.org/wiki/Separation_of_concerns).
00070 //!
00071 //!   Since the functionality of the policies are orthogonal to each other,
00072 //!   i.e., they do not depend on each other or their host (the differential
00073 //!   equation class), a Cartesian product of combinations are possible,
00074 //!   depending on which policies are selected. _This constructor registers all
00075 //!   possible combinations of policies for all available differential
00076 //!   equations._ By _register_, we mean, an entry is recorded in an associative
00077 //!   container, a std::map, that associates a lightweight key of type
00078 //!   walker::ctr::DiffEqKey, consisting of only an enum for each policy type,
00079 //!   to an std::function object that holds the a constructor bound to its
00080 //!   arguments corresponding to a particular differential equation + policies
00081 //!   combination. Note that registering these entries in the map does not
00082 //!   invoke the constructors. The mapped value simply stores how the
00083 //!   constructors should be invoked at a later time. At some point later,
00084 //!   based on user input, we then instantiate only the differential equations
00085 //!   (and only those configurations) that are requested by the user.
00086 //!
00087 //!   Since all differential equation types (registered in the factory)
00088 //!   "inherit" from a common "base", client-code is unform and generic, and
00089 //!   thus immune to changes in the inner workings of the particular
00090 //!   differential equations as long as they fullfill certain concepts, i.e.,
00091 //!   implement certain member functinos, enforced by the _common base_, DiffEq.
00092 //!   The words "inherit and "base" are quoted here, because the common base
00093 //!   does not use inheritance in the normal OOP sense and does not use
00094 //!   reference semantics, i.e., pointers, visible to client-code either. The
00095 //!   relationship is more of a _models a_-type, which simplifies client-code
00096 //!   and allows for the benfits of runtime inheritance with value-semantics
00097 //!   which is less error prone and easier to read. See more about the
00098 //!   _models-a_ relationship and its implementation in DiffEq/DiffEq.h.
00099 //!
00100 //!   The design discussed above allows the registration, instantiation, and
00101 //!   use of the differential equations to be generic, which eliminates a lot of
00102 //!   boiler-plate code and makes client-code uniform.
00103 //!
00104 //!   _Details of registration using mpl::cartesian_product:_
00105 //!
00106 //!   The template argument to mpl::cartesian_product requires a sequence of
00107 //!   sequences of types. We use vector of vectors of types, listing all
00108 //!   possible policies. The constructor argument to mpl::cartesian_product is a
00109 //!   functor that is to be applied to all combinations. mpl::cartesian_product
00110 //!   will then create all possible combinations of these types and call the
00111 //!   user-supplied functor with each type of the created sequence as a template
00112 //!   parameter. The user-supplied functor here is registerDiffEq, which, i.e.,
00113 //!   its constructor call, needs a single template argument, a class templated
00114 //!   on policy classes. This is the differential equation class to be
00115 //!   configured by selecting policies and to be registered. The arguments to
00116 //!   registerDiffEq's constructor are the factory, the enum denoting the
00117 //!   differential equation type, and a reference to a variable of type
00118 //!   std::set< ctr::DiffEqType >, which is only used internally to DiffEqStack
00119 //!   for counting up the number of unique differential equation types
00120 //!   registered, used for diagnostics purposes.
00121 //! \author J. Bakosi
00122 //******************************************************************************

00242   std::map< ctr::DiffEqType, std::size_t > cnt; // count DiffEqs per type

00245   for (const auto& d : g_inputdeck.get< tag::selected, tag::diffeq >()) {

00247       diffeqs.push_back( createDiffEq< tag::dirichlet >( d, cnt ) );
00248     else if (d == ctr::DiffEqType::GENDIR)
00249       diffeqs.push_back( createDiffEq< tag::gendir >( d, cnt ) );
00250     else if (d == ctr::DiffEqType::WRIGHTFISHER)
00251       diffeqs.push_back( createDiffEq< tag::wrightfisher >( d, cnt ) );
00252     else if (d == ctr::DiffEqType::OU)
00253       diffeqs.push_back( createDiffEq< tag::ou >( d, cnt ) );
00254     else if (d == ctr::DiffEqType::DIAG_OU)
00255       diffeqs.push_back( createDiffEq< tag::diagou >( d, cnt ) );
00256     else if (d == ctr::DiffEqType::BETA)
00257       diffeqs.push_back( createDiffEq< tag::beta >( d, cnt ) );
00258     else if (d == ctr::DiffEqType::NUMFRACBETA)
