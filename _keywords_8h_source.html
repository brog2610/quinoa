7  \brief Definition of all keywords
8  \details This file contains the definition of all keywords, including those
9  of command-line argument parsers as well as input, i.e., control, file
10  parsers. All keywords are shared among all parsers of all executables.
11 
12  All keywords are case-sensitive.
13 
14  The information contained in this file is used to build data structures for
15  on-screen help on the command-line arguments and control file keywords,
16  available via the --help, --helpctr, and --helpkw command-line arguments.
17 
18  A note on design: Defining structs that have static member functions
19  returning a std::string is a way of storing C++-style strings at
20  compile-time (which is not possible in a straightforward manner at this
21  time). This could also be done with C-style const char* as well. The
22  '*_info' structs store these strings, which then is used to specialize the
23  _kw::keyword_ template, defined in Control/Keyword.h. Specializing the
24  _keyword_ template also requires a specification of the precise string of
25  characters that make up a keyword eventually matched by the parsers. Since
26  these are all template arguments, the construction of keywords, their help,
27  as well as all grammars, are entirely assembled at compile-time. Since the
28  '*_info' struct member functions are static, they can be called without
29  instantiating an object and thus available at compile-time.
30 
31  The definition of an '*_info' struct _requires_ at least the name, short,
32  and long descriptions, defined by member functions _name()_,
33  _shortDescription()_, and _longDescription()_, respectively. Everything else
34  is optional. However, when adding a new keyword it is highly recommended to
35  define all of the _optional_ members if they make sense for the given
36  keyword. If an expect value type is also given, that can be hooked up into
37  where it is used.
38 
39  The most general definition of a keyword is as follows:
40 
41  \code{.cpp}
42  // Keyword info definition
43  struct keyword_info {
44 
45  // Required very short name, usually a single word or (for e.g.
46  // policies) a single character. This can be the keyword itself, but
47  // does not have to be. This field is used as an id of the option or
48  // setting.
49  static std::string name() { return "Name"; }
50 
51  // Required short keyword description
52  static std::string shortDescription() { return "Short description"; }
53 
54  // Required detailed keyword description. This returns a string literal,
55  // since this is usually multi-line and it is less work to maintain this
56  // way.
57  static std::string longDescription() { return
58  R"(Longer, possibly multi-line description of the keyword. Example
59  usage of the keyword is welcome here. Don't worry about formatting:
60  when this field is printed, extra spaces will be removed and line
61  breaks will be inserted.)";
62  }
63 
64  // Optional keyword alias. See also kw::Alias and
65  // <tpl_build>/pegtl/src/pegtl/include/pegtl/constants.hh for examples
66  // of what can be passed as template arguments (basically single
67  // characters). The one below defines the character 'c' as the alias.
68  // Aliases are single character long. The idea of an alias is to have a
69  // long as well as a short keyword for the same functionality.
70  // Currently, this is only hooked up for command-line arguments and not
71  // for control-file keywords, which is intentional. Command-line
72  // arguments are a lot less than control file keywords and are more
73  // frequently typed by the user. Thus command-line argument aliases are
74  // user-friendly. There are many control file keywords and aliases would
75  // only cause confusion. Defining an alias for a command-line argument
76  // enables the command-line parser to match on '--longer_keyword' as
77  // well as on '-c'. Depending on whether the alias typedef is defined
78  // for a keyword or not, the correct grammar is automatically generated
79  // at compile-time, matching on both the longer keyword as well as on
80  // the alias. Defining an alias for a control file keyword can be done
81  // but has no effect in a control file parser.
82  using alias = Alias< c >;
83 
84  // Optional expected data for the keyword - bundled to struct expect.
85  // This struct is entirely optional within a keyword definition.
86  // However, if it is defined, it must at least define the static member
87  // function description() which returns the description of the type the
88  // keyword expects. It may also optionally define the following fields:
89  //
90  // - type - defining the expected type
91  // - lower - lower bound of the expected value
92  // - upper - upper bound of the expected value
93  // - choices - valid choices for the expected value
94  //
95  struct expect {
96 
97  // If this struct is defined, required expected type description, max
98  // 10 characters long
99  static std::string description() { return "int"; }
100 
101  // Optional expected type
102  using type = std::streamsize;
103 
104  // Optional expected value lower bound
105  static const type lower = 1;
106 
107  // Optional expected value upper bound
108  static const type upper = std::numeric_limits< tk::real >::digits10 + 1;
109 
110  // Optional expected valid choices description, here giving
111  // information on the expected type and the valid bounds. Note that
112  // this can be any string, but if they exist, it is a good idea give
113  // at least some information on the bounds, as below, since the bounds
114  // are NOT displayed in the help for a keyword. This decision keeps
115  // the bounds specifications generic since they can be any type. As a
116  // result, the help structures, defined in HelpFactory.h, are simpler
117  // as they do not have to be parameterized by the type of the bounds,
118  // which greatly simplifies that code.
119  static std::string choices() {
120  return "integer [" + std::to_string(lower) + "..." +
121  std::to_string(upper) + ']';
122  }
123 
124  };
125 
126  };
127 
128  // Keyword definition, passing the above info struct as the first
129  // template argument, and the rest of the template arguments are the
130  // characters of the keyword to be matched by the parser. Remember: all
131  // keywords are case-sensitive. This one contrived example below defines
132  // keyword 'kw', matching the keyword 'KeYwOrD'.
133  using kw = keyword< keyword_info, K,e,Y,w,O,r,D >;
134  \endcode
135  \see Control/Keyword.h
136  \see Control/HelpFactory.h
137 */
