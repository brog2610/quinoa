00002 /*!
00003   \file      src/Control/Keywords.h
00004   \author    J. Bakosi
00005   \date      Wed 01 Jul 2015 02:25:56 PM MDT
00006   \copyright 2012-2015, Jozsef Bakosi.
00007   \brief     Definition of all keywords
00008   \details   This file contains the definition of all keywords, including those
00009     of command-line argument parsers as well as input, i.e., control, file
00010     parsers. All keywords are shared among all parsers of all executables.
00011 
00012     All keywords are case-sensitive.
00013 
00014     The information contained in this file is used to build data structures for
00015     on-screen help on the command-line arguments and control file keywords,
00016     available via the --help, --helpctr, and --helpkw command-line arguments.
00017 
00018     A note on design: Defining structs that have static member functions
00019     returning a std::string is a way of storing C++-style strings at
00020     compile-time (which is not possible in a straightforward manner at this
00021     time). This could also be done with C-style const char* as well. The
00022     '*_info' structs store these strings, which then is used to specialize the
00023     _kw::keyword_ template, defined in Control/Keyword.h. Specializing the
00024     _keyword_ template also requires a specification of the precise string of
00025     characters that make up a keyword eventually matched by the parsers. Since
00026     these are all template arguments, the construction of keywords, their help,
00027     as well as all grammars, are entirely assembled at compile-time. Since the
00028     '*_info' struct member functions are static, they can be called without
00029     instantiating an object and thus available at compile-time.
00030 
00031     The definition of an '*_info' struct _requires_ at least the name, short,
00032     and long descriptions, defined by member functions _name(),
00033     _shortDescription()_, and _longDescription()_, respectively. Everything else
00034     is optional. However, when adding a new keyword it is highly recommended to
00035     define all of the _optional_ members if they make sense for the given
00036     keyword. If an expect value type is also given, that can be hooked up into
00037     where it is used.
00038 
00039     The most general definition of a keyword is as follows:
00040 
00041     \code{.cpp}
00042       // Keyword info definition
00043       struct keyword_info {
00044 
00045         // Required very short name, usually a single word or (for e.g.
00046         // policies) a single character. This can be the keyword itself, but
00047         // does not have to be. This field is used as an id of the option or
00048         // setting.
00049         static std::string name() { return "Name"; }
00050 
00051         // Required short keyword description
00052         static std::string shortDescription() { return "Short description"; }
00053 
00054         // Required detailed keyword description. This returns a string literal,
00055         // since this is usually multi-line and it is less work to maintain this
00056         // way.
00057         static std::string longDescription() { return
00058           R"(Longer, possibly multi-line description of the keyword. Example
00059           usage of the keyword is welcome here. Don't worry about formatting:
00060           when this field is printed, extra spaces will be removed and line
00061           breaks will be inserted.)";
00062         }
00063 
00064         // Optional keyword alias. See also kw::Alias and
00065         // <tpl_build>/pegtl/src/pegtl/include/pegtl/constants.hh for examples
00066         // of what can be passed as template arguments (basically single
00067         // characters). The one below defines the character 'c' as the alias.
00068         // Aliases are single character long. The idea of an alias is to have a
00069         // long as well as a short keyword for the same functionality.
00070         // Currently, this is only hooked up for command-line arguments and not
00071         // for control-file keywords, which is intentional. Command-line
00072         // arguments are a lot less than control file keywords and are more
00073         // frequently typed by the user. Thus command-line argument aliases are
00074         // user-friendly. There are many control file keywords and aliases only
00075         // cause confusion. Defining an alias for a command-line argument
00076         // enables the command-line parser to match on '--longer_keyword' as
00077         // well as on '-c'. Depending on whether the alias typedef is defined
00078         // for a keyword or not, the correct grammar is automatically generated
00079         // at compile-time, matching on both the longer keyword as well as on
00080         // the alias. Defining an alias for a control file keyword can be done
00081         // but has no effect in a control file parser.
00082         using alias = Alias< c >;
00083 
00084         // Optional expected data for the keyword - bundled to struct expect.
00085         // This struct is entirely optional within a keyword definition.
00086         // However, if it is defined, it must at least define the static member
00087         // function description() which returns the description of the type the
00088         // keyword expects. It may also optionally define the following fields:
00089         // 
00090         //    - type - defining the expected type
00091         //    - lower - lower bound of the expected value
00092         //    - upper - upper bound of the expected value
00093         //    - choices - valid choices for the expected value
00094         //
00095         struct expect {
00096 
00097           // If this struct is defined, required expected type description, max
00098           // 10 characters long
00099           static std::string description() { return "int"; }
00100 
00101           // Optional expected type
00102           using type = std::streamsize;
00103 
00104           // Optional expected value lower bound
00105           static const type lower = 1;
00106 
00107           // Optional expected value upper bound
00108           static const type upper = std::numeric_limits< tk::real >::digits10 + 1;
00109 
00110           // Optional expected valid choices description, here giving
00111           // information on the expected type and the valid bounds. Note that
00112           // this can be any string, but if they exist, it is a good idea give
00113           // at least some information on the bounds, as below, since the bounds
00114           // are NOT displayed in the help for a keyword. This decision keeps
00115           // the bounds specifications generic since they can be any type. As a
00116           // result, the help structures, defined in HelpFactory.h, are simpler
00117           // as they do not have to be parameterized by the type of the bounds,
00118           // which greatly simplifies that code.
00119           static std::string choices() {
00120             return "integer [" + std::to_string(lower) + "..." +
00121                    std::to_string(upper) + ']';
00122           }
00123 
00124         };
00125 
00126       };
00127 
00128       // Keyword definition, passing the above info struct as the first
00129       // template argument, and the rest of the template arguments are the
00130       // characters of the keyword to be matched by the parser. Remember: all
00131       // keywords are case-sensitive. This one contrived example below defines
00132       // keyword 'kw', matching the keyword 'KeYwOrD'.
00133       using kw = keyword< keyword_info, K,e,Y,w,O,r,D >;
00134     \endcode
00135   \see Control/Keyword.h
00136   \see Control/HelpFactory.h
00137 */
